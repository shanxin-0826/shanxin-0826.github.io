Git

配置：​

name

    git config --global user.name "shanbaiwan"

email

    git config --global user.email "邮箱"

使用git：​

查看当前仓库的状态：

    git status;

初始化仓库：

    git init;

文件状态​

1.git中的文件有两种状态，未跟踪和已跟踪;

未跟踪指文件没有被git所管理，已跟踪指文件已被git管理;

2.已跟踪的文件又有三种状态：暂存，未修改，已修改;

暂存：表示文件修改已保存，但是尚未提交到git仓库;

未修改：表示磁盘中的文件和git仓库中文件相同，没有修改;

已修改：表示磁盘中文件已被修改，和git仓库中文件不同;

3.刚刚添加到项目中的文件处于未跟踪状态;

4.未跟踪 ---> 暂存：

    (1).将文件切换到暂存的状态：

        git add <filename>

    (2).将所有已修改(未跟踪)的文件暂存：

        git add *

5.暂存 ---> 未修改：

    (1).将暂存的文件存储到仓库中：

        git commit -m "xxxx"

    (2).提交所有已修改的文件(未跟踪文件不提交)：

        git commit -a -m "xxxx"

6.未修改 ---> 修改：

    修改代码后，文件会变为修改状态;

7.查看日志：

    git.log

常用的命令：​

1.重置文件：

    (1).恢复文件：

        git restore <filename>

    (2).取消暂存状态：

        git restore --staged <filename>

2.删除文件：

    (1).删除文件：

        git rm <filename>

    (2).强制删除：

        git rm <filename> -f

3.移动文件：

    (1).移动文件 重命名文件：

        git mv from to

分支​

1.git在存储文件时;

每一次代码的提交都会创建一个与之对应的节点;

git就是通过一个一个的节点来记录代码状态的;

2.节点会构成一个树状结构;

树状结构就意味着这个树会存在分支;

3.默认情况下仓库只有一个分支，命名为master：

4.在使用git时，可以创建多个分支;

分支与分支之间互相独立;

在一个分支上修改代码不会影响其他的分支;

5.命令：

(1).查看当前分支：

    git branch;

(2).创建新的分支：

    git branch <branch name>

(3).删除分支：

    git branch -d <branch name>

(4).切换分支：

    git switch <branch name>

(5).创建并切换分支：

    git switch -c <branch name>

(6).合并分支：

    git merge <branch name>

    更新版本发生冲突需要自选;

6.在开发中，都是在自己的分支上编写代码;

代码编写完成后，再将自己的分支合并到主分支中;

变基​

1.命令：

    git rebase <rebase name>

2.在开发中除了通过merge来合并分之外;

还可以通过变基来完成分支的合并;

3.我们通过merge合并分支时;

在提交记录中会将所有的分支创建和合并的过程都显示出来;

这样当项目比较复杂，开发过程比较波折时;

我们必须要反复的创建 合并 删除分支;

这样一来将会使得我们的代码提交记录变的极为混乱;

4.原理（变基时发生了什么）：

(1).当我们发起变基时;

git会首先找到两条分支的最近的共同祖先;

(2).对比当前分支相对于祖先的历史提交;

并且将它们提取出来存储到一个临时文件中;

(3).将当前部分指向目标的基底;

(4).以当前基底开始，重新执行历史操作;

5.变基和merge对于合并分支来说最终的结果是一样的;

但是变基会使得代码的提交记录更整洁更清晰;

注意：大部分情况下合并和变基是可以互换的;

但如果分支已经提交给了远程仓库，那么这时尽量不要变基;

远程仓库（remote）​

1.目前我们对于git所有操作都是在本地进行的;

2.在开发中显然不能这样;

这时我们就需要一个远程的git仓库;

3.远程的git仓库和本地的本质没有什么区别;

不同点在于远程的仓库可以被多人同时访问使用;

方便我们协同开发;

4.在实际过程中;

git的服务器通常由公司搭建;

内部使用或是购买一些公共的私有git服务器;

5.我们学习阶段，直接使用一些开放的公共git仓库;

我们我们常用的库有两个：GitHub和Gitee(码云);

6.将本地库上传git：

(1).git remote add origin http://github.com/.../...;

(2).修改分支的名字为mian(非必要)：

git branch -m main;

(3).git push 将代码上传到服务器上：

git push -u origin main;

7.将本地库上传gitee：

(1).git remote add <name> http://gitee.com/.../...;

(2).git push -u <name> main;

远程库的操作和命令​

1.列出当前关联的远程库：

    git remote

2.关联远程仓库：

    git remote add <远程库名> <url>

3.删除远程库：

    git remote remove <远程库名>

4.向远程库推送代码，并和当前分支关联：

    git push -u <远程库名> <分支名>

    git push <远程库名> <本地分支>:<远程分支>

5.从远程库下载代码：

    git clone <url>

6.关联成功后可以直接 git push;

如果本地的版本低于远程库，push默认是推不上去的;

7.要想推送成功，必须先确保本地库和远程库的版本一致;

git fetch会从远程仓库下载所有代码;

但是它不会将代码和当前分支自动合并;

使用fetch拉取代码后，必须要手动对代码进行合并;

8.从服务器上拉取代码并自动合并：

    git pull

9.注意：

推送代码之前，一定要先从远程库中拉取最新的代码;

tag标签​

1.当头指针没有执行某个分支的头部时;

这种状态我们称为分离头指针（HEAD detached）;

2.分离头指针的状态下也可以操作代码;

但是这些操作不会出现在任何的分支上;

所以注意不要在分离头指针的状态下来操作仓库;

3.如果非得要回到后边的节点对代码进行操作;

则可以选择创建分支后再操作：

    git switch -c <分支名> <提交id>

4.可以为提交记录设置标签，设置标签后;

可以通过标签快速的识别出不同的开发节点：

(1).给当前的节点打标签：

    git tag <标签名>

(2).给指定的节点打标签：

    git tag <标签名> <提交id>

(3).把指定标签推送到远程仓库：

    git push <远程仓库> <标签名>

(4).推送所有标签到远程仓库：

    git push <远程仓库> --tags

(5).删除本地标签：

    git tag -d <标签名>

(6).删除远程仓库标签：

    git push <远程仓库> --delete <标签名>

gitignore​

默认情况下，git会监视项目中所有的内容;

但是有些内容，比如node_modules目录中的内容;

我们不希望它被git所管理;

我们可以在目录中添加一个 .gitignore文件;

来设置那些需要git忽略的文件;

GitHub静态页面​

1.在GitHub中，可以将自己的静态页面直接部署到GitHub中;

它会给我们提供一个地址;

使得我们的页面变成一个真正的网站可以供用户访问;

2.要求：

(1).静态页面的分支必须叫做：gh-pages;

(2).如果希望页面可以通过xxx.github.io访问;

则需要将库的名字配置为xxx.github.io;

docusaurus​

1.Facebook推出的开源的静态内容管理系统;

通过它可以快速的部署一个静态网站;

2.使用：

(1).网址：https://docusaurus.io/

(2).安装：

npx create-docusaurus@latest my-website classic

(3).启动项目：

npm start 或 yarn start

(4).构建项目：

npm run build 或 yarn build

(5).配置项目：

docusaurus.config.js 项目的配置文件

(6).部署项目：

yarn deploy

3.上传代码时，两个注意事项：

(1).在配置文件里设置deploymentBranch来指定分支;

(2).需要配置一个环境变量：GIT_USER : github的用户名;

4.添加页面：

在docusaurus框架中，页面分成三种：

    1.page;

    2.blog;

    3.doc;
